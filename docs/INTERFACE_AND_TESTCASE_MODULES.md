# 接口模块 与 用例模块 设计说明

> 本文档详细说明测试管理平台中两个核心业务大模块——**接口（API Module）** 与 **用例（Case）**——的设计理念、组织结构、概念边界以及它们之间的关系。
>
> **主要参考来源**：`AIO接口平台手册（接口_调试用例_用例_场景之间的关系）.pdf`（2026-03-30）。本文档是该手册的工程化整理，配合本仓库的前端组件路径与后端服务定位，便于研发查阅。
>
> **撰写动机**：平台中存在多个层次的"用例"概念（调试用例 / 测试用例 / 单接口用例 / 场景用例），仅看名字非常容易混淆。本文统一术语并标注落地位置。

```
                                                  ┌───────────────────────┐
   ┌─ 文件导入 ─┐                                  │  调试用例（接口 1..n） │
   │           │                                  │  开发使用             │
   ├─ URL 导入 ─┼──► 接口源 ──┐                ┌──►│  - 引用接口定义        │
   │           │             │                 │  │  - 自带前后置/断言      │
   └─ 代码导入 ─┘             │                 │  └───────────────────────┘

                  └────────────────┬────────┘   ├──┤  ├─ 手工用例           │
                                   │             │  │  └─ 复制自调试用例 ◄──┐
                                   │             │  └───────────────────────┘ │
                                   │             │                            │
                                   │             │  ┌───────────────────────┐ │
                                   │             └──►  场景                  │ │
                                   │                │  └─ 步骤（引用/复制 ──┘ │
                                   │                │                     测试用例）
                                   │                │                        │
                                   └────────────────┴────────────────────────┘

注：当前数据范围 = 单应用 + 当前用例库；后续会扩展到跨应用、跨用例库。
```

**这张图传递了 5 件最重要的事**：

1. **接口定义是一切的根**——所有调试用例、测试用例、场景都从它衍生
2. **接口定义本身不含前后置/断言**——前后置和断言长在"调试用例"和"测试用例"上
3. **调试用例（开发用）和测试用例（测试用）是两类不同的对象**，都"长在某条接口下"
4. **测试用例可由调试用例复制而来**（也只能复制，不能引用）
5. **场景是基于测试用例编排的多步骤流程**

## 二、术语统一（重要！）

| 名称 | 所在模块 | 谁维护 | 本质 | 是否包含前后置/断言 |
|-----|---------|-------|------|-------------------|
| **应用** (Application) | 接口模块 | 开发 | 接口的归属容器，对应被测系统/服务/产品线 | — |
| **接口定义** (API) | 接口模块 | 开发 | 接口的"形状"——Tag 目录、接口名、请求部分、响应；可被同步、可被打分、可被生成文档 | ❌ 不包含 |
| **调试用例** (Debug Case，UI 当前显示「引用TEST」，**手册建议改名为"调试用例"**) | 接口模块（接口下） | 开发 | 引用接口定义的可执行单元；用于覆盖多种响应/业务逻辑；每个接口默认带 1 个 | ✅ 由调试用例承载 |
| **用例库** (Case Library) | 用例模块 | 测试 | 测试用例的归属容器；按测试目标、版本、迭代等维度组织 | — |
| **测试用例** (Test Case，UI 显示「CASE」) | 用例模块（单接口树下） | 测试 | 测试人员维护的可执行用例；可由调试用例复制而来，也可手工/AI 新建 | ✅ 由测试用例承载 |
| **单接口用例** (Single-API Case) | 用例模块（单接口子 Tab） | 测试 | "测试用例"在单接口子 Tab 下的口语化叫法 | 同测试用例 |
| **场景** (Scene) | 用例模块（场景子 Tab） | 测试 | 多接口业务链路；以**步骤**编排测试用例/调试用例/接口 | — |
| **场景步骤** | 场景内 | — | 5 类：导入系统请求 / 自定义请求 / 循环控制器 / 条件控制器 / 等待时间 | — |

**关键防混淆**：

- **「调试用例」**（接口模块）和**「测试用例」**（用例模块）是**两个不同的实体**，不要再当成"同一对象的两个标签"。
- 但二者的字段结构都继承自**接口定义**，因此看起来很像。
- 二者通过 §6 的"导入"机制建立关联（复制 / 完全引用）。

**UI 标签速查**：

| UI 上看到的标签 | 实际是什么 |
|----------------|-----------|
| `引用TEST`（紫色） | 接口模块下的**调试用例**（手册建议改名"调试用例"） |
| `CASE`（紫色） | 用例模块下的**测试用例** |
| `场景`（紫色） | 用例模块下的**场景** |
| `导入请求`（步骤前缀） | 场景中"导入系统请求"步骤的来源（接口/调试用例/测试用例任一种） |

## 三、关联模式：引用 / 复制 / 继承

平台内"对象 A 关联到对象 B"有 **3 种模式**：

| 模式 | 别名 / UI 提示 | 含义 | 可否编辑 | 与源解耦？ |
|-----|---------------|------|---------|-----------|
| **引用** (Reference) | "完全复制不能修改" / `引用 (不可编辑)` | 活引用，源端变化时这边自动跟随 | ❌ 不可编辑 | ❌ 强耦合 |
| **复制** (Copy) | "复制可以修改" / `复制 (可编辑)` | 独立快照，与源解耦 | ✅ 可编辑（接口地址通常被锁） | ✅ 解耦 |
| **继承** (Inherit) | "暂未排期" | 设计中的第三种关联模式（细节未定） | — | — |

**核心规则（PDF 明示）**：

- **引用 → 复制 单向一次性切换**：可以对引用过来的对象做"切换为复制"操作，**一次性脱离关系**；反向不允许。
- **被引用的对象不能删除**：例如接口定义如果被单接口用例引用、或者它下面有子调试用例，则**不允许删除接口**。
- **接口定义改动会同步到引用方**：如果接口定义新增/删除请求体字段，**所有引用它的调试用例和测试用例的字段集会同步更新**。

## 四、接口模块（API Module）

### 4.1 模块定位

> **核心定位**：接口模块**主要面向开发**——做接口的**维护**和**调试**。完成对接口及调试用例的维护功能，增强接口基本信息。

接口模块**不**承载业务测试用例与场景编排，那些归用例模块。

**菜单 / 路由 / 权限**：

- 一级菜单："接口"
- 列表路由：`/api-module/index`，权限码 `APP_API`（应用卡片列表）
- 详情路由：`/api-module/detail?id=<应用id>&protocol=HTTP`（隐藏，从列表卡片进入）

### 4.2 顶层容器：「应用」

「应用」是接口模块的顶层容器，对应一个被测系统/服务/产品线（例如示例截图中的"云层"、"电商2.0"）。

- 应用 + 协议（HTTP / MQTT / WebSocket / 协议驱动）共同决定一棵接口树
- 跨模块导入时，**应用是数据的源单位**（"导入单接口用例"弹窗第一步要选应用）
- 当前所有功能都以**单应用**为数据范围；**跨应用、跨用例库的能力是后续优化方向**

### 4.3 接口的两件事：定义 + 调试用例

**接口本身只承载「定义」**，不带前后置/断言：

| 维度 | 内容 |
|-----|-----|
| 基本信息 | Tag 目录、接口名、HTTP 方法、URL、所属环境 |
| 请求 | Query / Body（none / form-data / x-www-form-urlencoded / json / xml / raw）/ Header / REST 参数 |
| 响应 | 定义层面的响应结构（不是某次执行的实际响应） |
| 接口信息 | 元数据（创建人、修改时间、备注、Tag） |

**前后置 / 后置 / 断言落在调试用例上**（详见 §4.5）。

### 4.4 接口导入

支持 3 种导入：

- **文件导入**（常见格式）
- **URL 导入**
- **代码导入**（特殊场景）

**JMX 导入特殊规则**：

| JMX 内容 | 导入结果 |
|---------|---------|
| 单 http sample | 1 个接口 + 1 个调试用例 |
| 多 http sample，访问**同一**接口 | 1 个接口 + **多**个调试用例 |
| 多 http sample，访问**不同**接口 | 多个接口，各自匹配对应的调试用例 |

### 4.5 接口维护

> **核心能力**：实现对接口文档的解析得到完整的接口基本信息（**Tag 目录、接口名、接口的请求部分、响应**），可以额外维护接口信息。

- **同步语义**：通过接口维护实现该接口最底层的维护定义；所有引用用例都跟着定义走。复制的时候也先以接口来作为基础。**如果接口添加或删除请求体，引用它的调试用例和单接口用例中的定义同步**。
- **可手动维护**：包括优化请求字段、增加默认值及说明信息。
- **不能在接口维护中管理前后置/断言**——这些归调试用例。
- **删除约束**：接口定义如果被单接口用例引用或者有子调试用例，**不能被删除**。

### 4.6 调试用例（即"引用TEST"，PDF 建议改名为"调试用例"）

> **定义**：调试用例**引用**接口定义；每一个接口**一定有一个**调试用例（除非是手动新增的接口，默认不带调试用例）。

**职责与字段**：

- 包含接口定义中的**全部**内容（请求信息、前后置、断言等）
- 还可以**手动增加** Header 字段、额外的前后置/断言
- 可以执行（"发送请求"）、可以选择环境
- **不可修改**所引用的接口定义内容（接口定义那边的字段在调试用例里灰色不可编辑，仅能加自己的）

**为什么一个接口可以有多个调试用例？**

- 覆盖**多个响应情况**（成功 / 业务失败 / 鉴权失败 / 边界异常等）
- 处理**响应规则相同但业务逻辑不同**的输入

**典型例子**：

```
POST /login    ← 接口定义
 ├─ 引用TEST  200 用户Login成功
 ├─ 引用TEST  200 管理员Login成功
 └─ 引用TEST  422 用户Login失败
```

**环境策略**：调试用例**无需保存环境信息**——开发可以手工输入 ip + 端口直接运行；只有数据库等信息**必须**走环境配置。

**未来规划（PDF 提示）**：调试用例后期会成为**知识图谱**的一部分。能进入知识图谱的是"固化调试用例"，可被引用，作为开发配置的某些业务特性模板参考。

### 4.7 接口打分（AI）

- 在接口详情页右上角点击「AI 接口打分」即可
- 维度（举例）：接口基本信息完整性、请求参数完善度、请求体内容质量、响应定义完整性
- 总分 0~100，等级 A/B/C/...，风险等级标识
- **打分历史**：可查看 5 次以内的历史评分，对比变化
- **未来规划**：因为接口有同步功能，会存在"源版本"与"AIO 维护版本"两层概念，打分会和版本挂钩——支持了解导入时接口的评分、AIO 维护后接口的评分、同步导致接口变化带来的打分。

### 4.8 文档

- 将接口定义的内容转化为文档格式
- 每个接口需要显示对应的打分内容
- **支持调试用例的文档化显示**（请求头、Query 参数、REST 参数、请求体等都列入文档）
- 支持导出
- 实现完整的所有接口的文档化管理便于导出

### 4.9 Mock

- 接口详情页 `Mock` Tab：基于本接口生成 Mock 服务的配置（细节待补）

### 4.10 执行记录

- 记录所有执行信息——**接口、调试用例、用例、场景的执行都汇总在这里**
- 支持过滤
- 支持执行结果的简单 AI 分析（参考报告部分）

### 4.11 接口树的组织

接口树可任意层级，节点类型：

| 节点 | UI 标识 | 角色 |
|-----|---------|------|
| 目录 / 服务 | 📁 | 任意嵌套，按业务域或微服务分组 |
| 回收站 | 🗑 | 软删归档 |
| 接口（API） | HTTP 方法色块 | 接口定义 |
| 调试用例 | 接口节点子项，前缀紫色 `引用TEST` 标签 | 接口定义的调试单元 |

**「副本」标识**：接口名带 `(副本)` 后缀的是接口复制功能产生的；可多次复制（出现 `(副本)(副本)`）。

## 五、用例模块（Case Module）

### 5.1 模块定位

> **核心定位**：用例模块**主要面向测试**——完成测试人员设计**测试用例 / 场景**、执行测试的功能模块。
>
> 与接口模块最大的区别：**用例模块给测试用，接口模块给开发用**（接口模块是开发维护接口及测试接口的调试功能）。

### 5.2 层级结构

```
项目 (Project)
 └── 用例库 (Case Library)            ← /case/detail?id=<libraryId>
      ├── 功能 Tab                     （传统功能用例 + 模糊用例）
      ├── 接口 Tab                     ← 与接口模块关联最紧
      │    ├── 单接口        （CASE 列表 + 树）
      │    ├── 场景          （多步骤业务链路）
      │    ├── 测试计划       （cron 调度 + 多脚本/场景）
      │    ├── 测试报告       （场景执行报告）
      │    └── 定时任务       （所有定时任务汇总）
      ├── UI Tab                       （UI 自动化）
      └── 知识库 Tab
```

**菜单 / 路由 / 权限**：

- 一级菜单："用例"
- 列表路由：`/case/index`，权限码 `APP_CASE`
- 详情路由：`/case/detail?id=<用例库id>`（隐藏）
- 编辑路由：`/case/edit`（隐藏）

### 5.3 单接口子 Tab（重点）

「单接口」是用例库内**针对单一接口的测试用例**集合，节点类型：

| 节点 | UI 标识 | 来源 |
|-----|---------|------|
| 目录 | 📁 | 用户自建 |
| 回收站 | 🗑 | 软删归档 |
| 接口（API） | HTTP 方法色块 | 从接口模块导入（引用 or 复制）；复制的会进入"自定义接口"目录 |
| **测试用例 (CASE)** | 接口节点子项，前缀紫色 `CASE` 标签 | 从接口模块的调试用例**复制**而来，或手工/AI 新建 |
| 自定义接口（特殊目录） | 📁「自定义接口」 | **复制过来的接口的统一归宿**——因为它已经脱离接口定义管理 |

**树工具按钮（树顶部 「+」）**：新建目录 / 新增接口用例 / 刷新 / 折叠 / 导入 / 导出。

#### 5.3.1 测试用例的两种形态

| 形态 | 来源 | 可编辑性 |
|-----|-----|---------|
| **复制的普通用例** | 复制自调试用例、手工新建、AI 生成、另存为副本 | 内容可改可删（**接口地址被锁定**） |
| **完全引用调试用例的测试用例** | 在导入时选择"引用"模式（仅在场景里支持，单接口子 Tab 实际只支持复制） | **只能改用例名**，其他全锁 |

> 在测试用例中**无法进行接口维护操作**（如打分），上级接口仅做展示。

#### 5.3.2 新增接口用例（手动）

3 步流程：

1. 定义用例格式
2. 生成定义的接口及下属用例
3. **放入"自定义接口"** 目录

> 主要为测试手动添加测试用例服务（**可以不是接口维护范围内的**）。

#### 5.3.3 AI 生成单接口用例

入口：在接口上点击「AI 生成用例」→ 弹窗「AI 生成测试用例」。

**用例分类与默认优先级**：

| 大类 | 子类（→ 默认优先级） |
|-----|---------------------|
| **正向** | 仅传必要字段 → P0；语义符合要求 → P0；覆盖枚举组合 → P1；其他正向 → P2 |
| **反向** | 缺失必填字段 → P1；语义非法 → P1；无效值 → P2；格式错误 → P2；类型错误 → P2；其他反向 → P3 |
| **边界值** | 超出最大、最小边界值 → P1；极大值/极小值 → P1；Null/零值/空值 → P2；字符串过长、过短 → P2 |
| **安全性** | 鉴权控制 → P0；SQL注入 / XSS注入 / 命令行注入 / JSON注入 / NoSQL注入 → P1；模糊输入 → P3 |
| **自定义提示词（其它）** | 用户填自由提示词 → P3 归为其他类型 |

**机制**：

> 无论是 AI 生成用例还是添加用例**都是复制用例**，以接口定义为基础，AI 会使用规则对输入进行填充，并自动附上对应的断言策略。

**新建后的位置**：新建的用例会在**接口和目录分类下**看到。

#### 5.3.4 另存用例

> 当前用例可以被执行也可以被另存（复制），从而克隆一个一样的用例，名字附加"副本"结尾。

入口：用例详情页「保存」按钮旁的下拉 → 「另存为用例」 / 「删除」。

#### 5.3.5 导出

支持用例的导出，支持常见语法结构例如 pytest、junit 等（细节略）。

### 5.4 功能 / UI / 知识库 Tab

## 六、两个模块的关联关系

### 6.1 数据流向

接口模块产生**接口定义 + 调试用例**，用例模块通过"导入"或自建产生**测试用例**。两者通过下面 3 个入口建立联系：

| 入口 | 位置 | 作用 |
|-----|-----|------|
| **导入单接口用例**弹窗 | 用例库 → 接口 Tab → 单接口 → 树 + 「导入」 | 从接口模块拉接口定义/调试用例到本用例库 |
| **导入系统请求**弹窗 | 用例库 → 接口 Tab → 场景 → 步骤 + 「导入系统请求」 | 从接口模块/本用例库拉接口/调试用例/测试用例进场景作为步骤 |
| **同步**机制 | 后台逻辑 | 接口定义变更时，引用方（调试用例 / 测试用例）的字段集自动同步 |

### 6.2 「导入单接口用例」弹窗（在用例库的单接口子 Tab）

```
┌──────────────────────────────────────────────────────┐
│  导入单接口用例                              [×]    │
├──────────────────────────────────────────────────────┤
│  选择应用：[ 电商 2.0                        ▾ ]    │
│  ┌─────── API ───────┬─────── CASE ───────┐          │
│  │ 🔍 搜索接口名称或路径                              │
│  │ ☐ GET   Get Order             /orders/{order_id} │
│  │ ☐ GET   Health Check          /health            │
│  │ ☐ POST  Register              /register          │
│  │   …                                                │
│  └────────────────────────────────────────────────────┘
│                              [取消]   [确定]         │
└──────────────────────────────────────────────────────┘
```

**两个 Tab**：

- **API Tab**：列出所选应用下的**接口定义**——可选「**引用**」或「**复制**」模式
- **CASE Tab**：列出所选应用下的**调试用例**——**只能复制**（PDF 明示："可以从接口定义中导入接口（引用/复制）或者调试用例（复制）"）

**导入后的形态**：

| 选了什么 + 模式 | 用例库里出现什么 |
|---------------|-----------------|
| API + 引用 | 接口节点（活引用，源改这边变） |
| API + 复制 | 接口节点（独立快照），归类到「**自定义接口**」目录 |
| CASE + 复制 | 1 条测试用例 + 它依附的接口（如本库尚无该接口）；接口归到「自定义接口」 |

### 6.3 关联模式速查（接口/调试用例/测试用例 → 各种入口）

| 入口 | 接口（API） | 调试用例 | 测试用例 |
|-----|------------|---------|---------|
| 单接口子 Tab 导入 | 引用 / 复制 | 复制（只能） | — |
| 场景的"导入系统请求" | **强制引用**（接口是自定义步骤模式） | **强制完全引用** | 引用 / 复制（用例库下） |
| AI 生成场景时关联 | 用 AI 自动填数据 | 强制只使用调试用例内容 | 复制可 AI 替换数据 / 引用无法修改 |

## 七、场景（Scene）

### 7.1 场景定位

> **作为围绕业务的多接口链路称之为场景，支持 AI 生成**。

场景在用例库 → 接口 Tab → 「场景」子 Tab 下管理。

### 7.2 场景元数据

每个场景含：

- 名称、优先级（P0~P3）、状态（草稿 / ...）、负责人
- 步骤数
- Tab：用例信息 / 步骤 / 参数 / 前后置操作

### 7.3 场景步骤

**5 种步骤类型**（场景详情 → 步骤 Tab → 「+ 添加步骤」下拉）：

| 步骤类型 | 作用 |
|---------|------|
| **导入系统请求** | 引用/复制接口模块的接口、调试用例，或本用例库的测试用例 |
| **自定义请求** | 在场景里临时定义一个请求 |
| **循环控制器** | 重复执行子步骤 N 次 |
| **条件控制器** | 按条件选择执行分支 |
| **等待时间** | 等待 N ms |

### 7.4 「导入系统请求」弹窗

```
┌──────────────────────────────────────────────────────┐
│  导入系统请求                                [×]    │
├──────────────────────────────────────────────────────┤
│  选择应用：[ 电商 2.0                        ▾ ]    │
│  ┌─────── API ───────┬─────── CASE ───────┐          │
│  │  …接口/CASE 列表…                                  │
│  └────────────────────────────────────────────────────┘
│  关联方式：◉ 引用（不可编辑）  ○ 复制（可编辑）        │
│                              [取消]   [确定]         │
└──────────────────────────────────────────────────────┘
```

**关联规则（PDF 明示）**：

- **接口**：自定义步骤模式 → 强制是**引用**
- **调试用例**：**完全引用**（只能用调试用例内容）
- **用例库下的测试用例**：可**引用**（用例变化这里步骤也变化）/ 可**复制**（独立维护）

**当前限制**：

> 现在只能对**当前用例库**来做过滤，**不支持跨应用跨用例库**概念（后续优化）。

### 7.5 AI 生成场景

**3 步交互**：

1. **输入场景描述**：自然语言（例如"登录、查询、下单"），可上传 RAG 文件辅助
2. **确定步骤**：AI 进行步骤解析，列出步骤名（可手动加/改名，影响检索匹配率）
3. **指定步骤使用接口**：AI 列出匹配定义接口下当前测试用例中的所有用例，并附**匹配率**——用户可调整或手动关联接口

**关键实现细节**（PDF 注解）：

> 通过查看接口信息（**这里需要优化给出更加清晰的接口信息**，例如接口的名称、接口下的调试用例，提供给用户了解应该用什么接口及对应的调试用例模板作为基础），AI 列出匹配定义接口下当前测试用例中的所有用例（可以给一个配置匹配率的选项），方便用户使用估计信息而不做 AI 优化。

> 如果选择具体的用例，则 AI 不针对该用例做任何 AI 辅助操作（修改 Header、添加后置操作等）。

> 现实匹配的接口步骤，如果没有匹配到合适的接口可以手动关联接口。

**底层规则**：

- 选**接口**：AI 自动填数据
- 选**测试用例**：按用例内容填写；复制时可 AI 替换数据；引用时无法修改
- 选**调试用例**：强制只使用调试用例内容

### 7.6 场景执行 / 调试

- 场景中可以实现**环境及环境集群**的管理，从而支持**单步调试**及**多步调试**
- 可以在场景列表实现对某个场景的**执行**工作
- 支持**定时任务**

## 八、测试计划 / 测试报告 / 定时任务 / 环境配置

### 8.1 测试计划

> **提供基于定时任务的多脚本/场景执行能力**，支持**并行/串行**执行模式、**cron** 计划任务、**参数化**等。

- 入口：用例库 → 接口 Tab → 「测试计划」子 Tab
- 计划基本信息：计划名、负责人、备注、创建时间、更新时间
- 执行模式：串行 / 并行
- 定时执行：开关 + cron
- 条目列表：可加入接口、场景、测试用例等条目，每条可设参数和定时

### 8.2 测试报告

针对场景执行的记录功能。可查看测试报告，了解信息详情。

**关键能力**：

| 能力 | 说明 |
|-----|------|
| **基础展示** | 通过率、总用例数、执行耗时、跨度时间；执行明细（步骤名、类型、状态、耗时） |
| **AI 报告分析** | 对整个场景执行的错误进行**分类统计**并给出修复建议 |
| **第三方日志接入** | 支持各类服务端 log 导入；AI 同步分析给出更详细的错误分析及修复建议（体现后台业务逻辑错误） |
| **代码映射分析** | 基于 AI 分析 + 接口由代码导入为基础，分析服务器日志后**在错误位置可以直接查看对应的代码文件内容**，提供更精确的错误分析能力 |

### 8.3 定时任务

> 列出当前所有定时任务内容（**场景和测试计划**）。

- 入口：用例库 → 接口 Tab → 「定时任务」子 Tab
- 与场景列表 / 测试计划列表内的定时入口互补：本子 Tab 是**汇总视图**

### 8.4 接口环境配置

> **针对应用维护接口所需要的 Http 地址、数据库、统一前后置等信息**。

- 支持**导入导出**
- 支持**信创数据库**
- 支持**环境变量**

> 注：调试用例**无需**保存环境信息（开发可手工填 ip+端口）；但**数据库等信息必须走环境配置**。

## 九、特别注意事项

> **PDF 原文摘录**（核心警示）：
>
> 接口定义是基础，**一旦接口定义变化会影响后续所有的调试用例、测试用例、场景**。接口定义后续会有**版本概念**，如果变更定义会**提示影响范围**。
>
> 当前都以**单应用、当前用例库**作为数据范围。后续处理跨应用、跨用例库情况。

实操要点：

1. **改接口定义前先评估影响**——它是所有引用方的"上游"，字段增删会同步到调试用例、测试用例和场景中（场景里"引用模式"下的步骤）。
2. **当前不支持跨应用、跨用例库引用**——所有数据范围在单应用 + 当前用例库内。需要跨域的能力请走"复制"模式作为临时方案。
3. **删除接口前确认无引用**——若接口有子调试用例或被单接口用例引用，删除会被拒绝。
4. **复制的接口归到"自定义接口"目录**——这些接口不再受接口定义管理，自行维护。
5. **测试用例中接口地址被锁**——只能改字段值，不能改接口地址；引用调试用例的测试用例则只能改用例名。
