# 用例参数 - CSV 导入与参数组重构需求

> 本文档描述「用例详情 - 参数 Tab」中 CSV 导入功能的重构需求，以及由此引入的「参数组（参数集）」概念。
>
> 仅描述需求与交互目标，不写技术实现方案；具体设计会在评审通过后另行落地。

## 1. 背景与现状

### 1.1 入口位置

- 路由：用例详情页（`src/views/case/detail/...`）下的「参数」Tab。
- 视图文件：`test-mng-web/src/views/case/detail/components/scene/SceneParameters.vue`。
- CSV 弹窗：`test-mng-web/src/views/case/detail/components/scene/CsvVariableDialog.vue`。
- 顶部「添加变量」下拉目前有两个选项：`常量`、`CSV`。

### 1.2 现有 CSV 弹窗的问题

当前 `CsvVariableDialog.vue` 把每个 CSV 文件当成「一个变量」来处理，弹窗暴露了大量用户根本不需要关心的低层细节：

1. 必填「变量名」输入框 —— 但 CSV 文件本身的表头列名就是参数名，让用户再起一个变量名既冗余又容易和列名冲突。
2. 强制选择 `Encoding`（UTF-8 / GB2312 / …）、`分隔符`、`是否允许带引号` —— 用户大多数情况下根本不知道自己 CSV 是什么编码，错选会直接导致解析乱码。
3. 「预览」是一个只读 textarea，把 CSV 原文贴出来，没办法直观看到「我导入后会得到几列、几行」。
4. 数据语义混乱：现在 CSV 变量被存成 `valueType=CSV` 的单条 `VariableItem`，但实际包含的是「多列 × 多行」的参数集合，没有结构化字段表达「列名 = 参数名、行 = 一组参数」这件事，导致后续执行时拿不到一份清晰的「参数组列表」。
5. 「添加变量」下拉中 `常量` 与 `CSV` 是同一层级的并列选项，但二者的语义其实是「一个变量」 vs「一组参数集」，不在同一抽象层。

### 1.3 现有数据结构（仅作参考）

目前 `Scene.VariableItem`（`src/api/interface/scene.ts`）通过同一个对象表达多种值类型（STRING / CSV / LIST / COUNTER / RANDOM 等）。CSV 走 `valueType=CSV`，并把 `filePath / csvPreview / encoding / delimiter / allowQuotes` 直接挂在 VariableItem 上。本次重构会重新讨论这个模型，不强制保留这些字段。

## 2. 重构目标

1. **降低 CSV 导入的认知成本**：用户只要选一个 CSV 文件，就能拿到结构化的参数组，不需要选编码、分隔符、引号策略，也不需要再起变量名。
2. **明确「参数组（参数集）」概念**：CSV 表示的是「多组参数」，要在数据模型、UI 文案、执行逻辑上一致地体现这一点。
3. **预览即所得**：导入后用户立刻看到一张表格（表头 = 参数名，每一行 = 一组参数），所见即所用。
4. **为后续「执行用例时按参数组循环」打基础**：参数组在用例执行时可以被一次取一组，或循环跑完所有组。

## 3. 详细需求

### 3.1 CSV 导入弹窗（重构后）

#### 3.1.1 弹窗结构（建议）

弹窗自上而下：

1. 标题：`导入参数（CSV）`
2. 说明文案（一段灰字提示）：

   > CSV 文件第一行作为参数名，每一行表示一组参数。例如：
   > 第一行 `username,password` 表示有两个参数；
   > 第二行 `alice,123456` 表示一组参数（alice / 123456）；
   > 第三行 `bob,123abc` 表示另一组参数。
   > 用例执行时会按行依次/循环取用这些参数组。

3. **「下载模板」按钮**：保留现有下载逻辑，但文案与位置调整。
   - 按钮文案改为 **「下载模板」**（不要叫「下载」，避免和「下载已上传文件」混淆）。
   - 模板内容举例：
     ```csv
     username,password
     alice,123456
     bob,123abc
     ```
   - 鼠标悬浮 tooltip：`下载示例模板，按格式填写后再上传`。

4. **文件上传区**：
   - 拖拽区 + 点击选择文件，仅接受 `.csv`。
   - 上传后显示文件名 + 删除按钮。
   - **不再展示**：Encoding 下拉、分隔符输入、是否允许带引号下拉。这三项由后端自动嗅探。

5. **解析预览区**（核心交互）：
   - 上传成功后，前端调用后端解析接口拿到结构化结果，渲染成 **表格**（不是 textarea）。
   - 表格表头 = CSV 第一行的列名（即参数名）。
   - 表格行 = CSV 数据行（即「参数组」）。每行左侧带行号（`第 1 组 / 第 2 组 …`）。
   - 行数较多时分页或固定高度滚动，默认显示前若干行 + 「共 N 组参数」。
   - 解析失败（编码/格式错误）时给出明确错误提示，不展示残缺表格。

6. **底部按钮**：`取消` / `确定`。
   - 确定时校验：必须已上传并解析成功。
   - 不再校验「变量名」（已删除该字段）。

#### 3.1.2 删除项清单（相比现状）

| 字段/控件 | 处理 | 原因 |
|----------|------|------|
| 变量名 输入框 | 删除 | CSV 表头就是参数名 |
| 描述 输入框 | **保留**，作为这一组参数集的备注 | 便于用户自己分辨多个参数集 |
| Encoding 下拉 | 删除 | 后端自动嗅探（UTF-8 / UTF-8 BOM / GBK / GB2312…） |
| 分隔符 输入 | 删除 | 后端自动嗅探（`,` / `;` / `\t`） |
| 是否允许带引号 | 删除 | 后端默认按 RFC 4180 处理 |
| 预览 textarea | 删除 | 改为表格预览 |

#### 3.1.3 新增项清单

| 字段/控件 | 说明 |
|----------|------|
| 参数集名称 | 用户给这一组参数集起个业务名（可选；不填时用文件名兜底，去掉 `.csv` 后缀）。该名称是「参数集」在变量列表里的展示名，不是某一列参数名 |
| 表格预览 | 上传后立即显示，表头 = 参数名列表，行 = 参数组 |
| 参数组数量提示 | 「共 N 组参数，每组 M 个参数」 |

### 3.2 后端解析接口（行为约定，不限定签名）

> 接口具体路径与字段在设计阶段确定。这里只描述行为契约。

- 入参：CSV 文件（multipart/form-data）。
- 行为：
  - 自动嗅探编码（优先级：BOM → UTF-8 → GBK / GB18030 → 失败提示）。
  - 自动嗅探分隔符（候选：`,` `;` `\t`，按出现频次与一致性投票）。
  - 按 RFC 4180 兼容引号、转义。
  - 校验：第一行不能为空、不能有重复列名、不能有空列名；数据行列数与表头列数必须一致。
- 返回：
  - `headers: string[]` —— 参数名列表（CSV 表头）。
  - `rows: string[][]` —— 每个元素是一组参数（与 `headers` 一一对应）。
  - `rowCount`、`columnCount`。
  - `filePath` —— 已落地的文件存储路径，便于后续执行复用。
- 失败：返回明确错误码 + 用户可读 message（例：「第 3 行列数与表头不一致」）。

### 3.3 数据模型调整：参数组（CsvParameterSet）

把「CSV 文件」从一个 `VariableItem` 升级为一个独立概念 —— **参数集 / 参数组列表**。

建议在变量数据上区分两类：

| 类型 | 语义 | 来源 |
|------|------|------|
| 常量变量（CONSTANT / STRING） | 一个 name → 一个 value | 用户在表格里手填 |
| 参数集（PARAM_SET，由 CSV 导入） | 一个集合，包含 N 个参数名 + M 行参数组 | CSV 导入弹窗 |

参数集需要持久化的字段（命名以设计阶段为准）：

- `id`
- `name` —— 参数集名称（用户填或文件名）
- `description` —— 备注
- `headers: string[]` —— 参数名列表
- `rows: string[][]` —— 所有参数组
- `filePath` —— 原 CSV 存储路径（便于回看、重新解析）
- `status` —— 启用 / 禁用
- 关联：所属用例 / 场景 ID

> 注意：现有 `Scene.VariableItem` 上的 `valueType=CSV` 写法、`filePath / csvPreview / encoding / delimiter / allowQuotes` 字段在重构后**不再使用**。前端若有兼容老数据的需要，做一次性迁移即可，不再保留双结构兼容代码（避免长期遗留分支）。

### 3.4 「添加变量」下拉调整

当前下拉：`常量` / `CSV`。重构后建议改为：

- **添加常量** —— 弹出一行可编辑的常量行（同今日「常量」行为）。
- **导入参数集（CSV）** —— 打开重构后的 CSV 导入弹窗。

文案明确表达「常量是一个变量」「参数集是一组参数」。「CSV」只是导入方式，不应作为下拉选项的主词。

### 3.5 参数 Tab 的列表展示

参数 Tab 主表格需要同时展示「常量」和「参数集」两类，但两类信息密度不同，建议：

- 常量行：照旧（变量名 / 变量值 / 描述 / 状态 / 操作）。
- 参数集行：
  - 「变量名」列展示参数集名称（带一个标签 `参数集` 区分常量）。
  - 「变量值」列展示 `查看`（点击打开预览弹窗，回到 3.1 的表格预览只读态）。
  - 副信息：`共 N 组 · M 个参数`。
  - 「描述」「状态」「操作（编辑/删除/重新上传）」与常量保持一致。

具体 UI 走查在视觉评审阶段定，本文档只确认「两类需在同一张表格里清晰区分」。

## 4. 执行阶段：参数组的使用方式（下一阶段讨论）

> 本节是后续讨论的占位章节。先把已经达成共识的几点列下来，避免在第一阶段就纠缠执行逻辑。

### 4.1 已达成的方向

1. **调试（单次执行）**：从参数集中取一组（默认第 1 组，可由用户指定第 N 组）注入到用例执行上下文。
2. **循环执行**：用例配置「按参数集循环」时，参数集每一行触发一次执行；总轮数 = 参数组行数。
   - 示例：参数集有 10 个用户，登录用例就跑 10 次，每次用一组。
3. **执行报告**：循环完成后产出汇总报告，包含每一轮的：
   - 使用的参数组（哪一行）。
   - 步骤通过/失败情况。
   - 失败原因摘要。
   - 整体通过率。

### 4.2 待讨论的问题

- 若一个用例引用多个参数集（例如 `用户参数集` × `订单参数集`），是「笛卡尔积」还是「按行对齐」？默认推荐：按行对齐，不齐时按短的截断 + 警告；笛卡尔积作为高级选项。
- 参数集为空 / 行数为 0 时，调试和循环行为分别是什么？建议：调试报错「参数集为空」，循环跳过整个用例并在报告中标记。
- 同名参数冲突：当前 Tab 中的常量与参数集表头同名时谁覆盖谁？建议：参数集优先（更具体），但要在 UI 上提示用户。
- 执行报告的入口、留存策略、和现有「场景执行结果」如何整合 —— 在执行阶段需求文档中单独描述。

## 5. 验收标准（第一阶段：导入与建模）

第一阶段只交付「CSV 导入 + 参数集建模 + 列表展示」，验收点：

1. 「添加变量」下拉文案与行为按 3.4 调整。
2. CSV 导入弹窗按 3.1 重构：无编码/分隔符/带引号/变量名字段；保留并改文案的「下载模板」；上传后调用后端解析，前端用表格展示。
3. 后端提供解析接口，按 3.2 行为契约工作；编码与分隔符自动识别正确率覆盖常见用例（UTF-8 / UTF-8 BOM / GBK，`,` / `;` / `\t`）。
4. 参数 Tab 主列表能区分「常量」和「参数集」两类，参数集行点击「查看」可重新预览表格。
5. 旧的 `valueType=CSV` 单变量数据有迁移方案，迁移后前端不再有兼容老结构的代码分支。
6. 第一阶段不引入执行侧改动，但数据结构需为第 4 节的执行需求预留好字段（`headers / rows / filePath`）。

## 6. 不在本阶段范围内

- 用例执行引擎对参数集的消费逻辑、循环驱动、报告生成（属于第二阶段）。
- 参数集的导出（导出回 CSV）。
- 参数集行级编辑（在表格里直接修改某一行某一列的值）—— 第一阶段以「重新上传 CSV」覆盖即可。
- 大文件分片上传与流式解析 —— 先以单文件 ≤ 10MB / ≤ 1 万行为约束，超出给出提示。
